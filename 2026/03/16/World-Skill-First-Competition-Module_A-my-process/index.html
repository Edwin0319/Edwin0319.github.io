Module A - Test Project

Web Technologies - Backend Development

  • Original Submitted by: Gilles GRANGER - FR
  • Updated by: Daniel Cheang - MO
  • Document: WSC2026_TP_MO

Introduction to Test Project

The FlexiWork company aims to provide a SaaS (Software as a Service) dedicated for coworking space managers. The main objective of this software is to manage allocation and booking of resources, such as desks and meeting rooms. In the long term, it should be highly flexible to adjust to a large variety of use cases.

However, to help them have a successful development with a product that really addresses the customer’s needs, the company partnered with a local coworking space to develop a pilot version, with a narrowed set of features. This minimal version should be usable on its own and allow the development team to collect continuous feedback and make quick iterations.

In this module, you will develop this v1. This version will be used by two types of users:

Coworking Space Administrator

The administrator will be able to:

  • Manage rooms: Create, update and access a list of meeting rooms
  • Manage users: View user’s profiles and give them credits they can use for different services
  • Get an overview of all rooms, with their associated bookings

Coworking Space User

A user will be able to:

  • List upcoming room bookings
  • Manage meeting room bookings: This includes searching for an available room and placing a booking. In this first version, the booking system will be simplified
  • Get a list of his previous transactions

Instructions to the Competitor

  1. No database schema is provided, you are responsible for creating it. Its quality will be assessed; this means you should avoid redundant data, use suitable indexes especially for foreign keys, use consistent namings for columns and table names, etc.

  2. You must include a file named database-dump.sql in your folder.

  3. Some fixtures are provided in Appendix 1. You must import them in your schema, ensuring data integrity. Be careful about typos!

  4. Security: You should also ensure that major security risks are addressed. This means the forms are secured, the sensitive data is properly hashed, and a user can only access the resources he’s supposed to access.

  5. Layout: The layout should be intuitive. No extra design effort is required, but all the pages and features should follow the common UX “laws”, so the user journey is simple to understand.

  6. Code Quality: You should keep in mind that the code will evolve frequently during the pilot phase, so you must apply code quality standards, so other developers can work on it.

  7. Testing: The app will be tested on Desktop mode only, on Google Chrome.

Description of Project and Tasks

Coworking Space Administrator

Login

An admin should be able to login with the following credentials:

  • Username: admin
  • Password: Very$ecret2026!
  • First Name: Super
  • Last Name: Admin

Upon login, an admin should view a navbar, with 3 links: “Home”, “Users” and “Rooms”. No registration is required, every account will be created in advance.

Home Page

The home page presents a schedule where the admin can see all the room bookings for a given day (by default, the current day). The active day is displayed on top of the schedule.

  • Each room is represented by one column, and rows represent hours
  • Each meeting room booking is displayed as a block, from the beginning time to the ending time
  • It is possible to see the previous or the next day by clicking on buttons on top of the schedule view
  • Each booking must display the user’s first and last names, and its beginning and ending times
  • Bookings don’t need to be clickable

Manage Rooms Page

An admin can access a list of the rooms in the coworking space.

  • In the list, each room should be displayed as a card, with its picture, name and capacity
  • All the cards should be clickable and bring the administrator on a page where he’ll be able to edit the room’s details
  • There’s also a “create room” button which shows a similar form, to create a new meeting room
  • No “delete room” action is required currently

When creating or updating a room, the following information should be filled (everything is mandatory, except for the picture when updating):

  • Name (unique)
  • Picture
  • Capacity (number between 2 and 100)
  • A “price” in credits, per hour of use

Optional services: There’s also a possibility to enable optional services the users can book at the same time (video projector, coffee machine, printer…). If enabled, the admin can set a price in credits that will be added to the booking price. The price can be 0, so the option is free for the user.

Important Notes:

  • If you update the price for the room or the options, or even remove an option that were available before, the bookings that have already been placed should not change and the users should see the exact amount they paid at the time they placed their booking, with all the options, even if their price or availability changed
  • The available options that can be linked to a room may evolve frequently in future versions. You can either store them in a clean way in your code, or in a separate table in the database, but you should avoid hardcoding it

Manage Users and View Transactions

The admin can list the users.

  • The list should be ordered by ascending last name and ascending first name if needed
  • No pagination is required
  • No admin user should be presented in this list
  • Each user is clickable, and its profile is displayed

The profile shows:

  • First name and last name of the user
  • A form to add or deduct credits to the user
  • The user’s credits balance, which is the sum of all the transactions listed below
  • A list of all transactions for the user:
    • All bookings, including the credits spent for the booking itself, and also showing the credits spent for each used option
    • All credits added by the admin

Transaction Ordering: All the transactions should be ordered by “created at” date. If a room booking is done on May 1st for June 30th, the transaction should appear on May 1st.

Coworking Space User

Login

Users should be able to login with one the following credentials:

User 1:

  • Username: user1
  • Password: $ecret2026!
  • First Name: John
  • Last Name: Doe

User 2:

  • Username: user2
  • Password: u$er2026!
  • First Name: Jane
  • Last Name: Doe

Upon login, a user should view a navbar, with 3 links: “Home”, “Search” and “Profile”. No registration is required, every account will be created in advance.

Homepage

On the homepage, a user should see the list of his upcoming bookings (for simplicity, you can consider that all the bookings for the current day are also upcoming).

This includes:

  • Room name
  • Total price (room booking and options), in credits
  • Remainder of the selected options
  • Date
  • Start/end time

Ordering: The bookings should be ordered by ascending date (the closest booking from the current date must come first).

Meeting Room Search and Booking

On the “Search” page, the user can view a form where the following information can be entered:

  • Booking date: Only one day. Multi-day bookings will be developed in a further version
  • Booking start and end times: Time slots are multiples of 1 hour. Every room can be booked from 8am to 7pm
  • Attendees: A number greater or equal to 1

Search Results: When submitting the form, all rooms where the capacity is equal or greater to the number of attendees are displayed. All the rooms are displayed as cards.

For each room, display:

  • Photo
  • Name
  • Price (number of hours * price per hour for the room)
  • Available options with their price

Availability Check: For each room, you must calculate if the room will be available between the start and end times. If another booking already exists and overlaps on the required time interval, then the room is unavailable.

Display Order: All available rooms should be shown before the unavailable rooms.

Status Indicators:

  • If the room is unavailable, it should be displayed in black and white, with a “Not available” tag
  • If the room is available for the whole day (no bookings on the whole day), a special “Available all day!” tag must be displayed

Booking Button: On available rooms, a “Book this room” button is displayed if the user has enough credits to book the room, without any options. Otherwise, the button must link to an external URL of your choice, and display “Buy Credits”.

Booking Interface

The user is now able to book the room, without changing the date and times. An interface looking like a shopping cart is displayed.

Main Block:

  • The date, room name, begin and end time are reminded
  • The user can select options in the main block. The options are the ones enabled for the room in the admin backoffice
  • The price for each option is displayed (it is fixed, and does not depend on the booking duration)

Side Block (Pricing): It evolves in real-time if the user checks an option:

  • The room base price is reminded: “X hours * Y credits/hr = Z credits”
  • For each option configured for the room, the user can check it. If checked, the option’s name and price is added to the cart
  • On the bottom of this side block, a “Confirm” button can confirm the booking. However, if after adding options, the user’s credit balance is too low, the button is disabled and displays “x missing credits” instead

Confirmation: Upon confirmation, the user’s credit balance is updated.

Profile Page

The user can see his own profile, in the exact same way as the admin views a user profile (identity, credit balance, transactions).

Difference: Instead of the form for adding credits, the user must see a single informative notice that says: “Coming soon: Top-up your credit balance online on this page!”
